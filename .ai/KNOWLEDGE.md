# Knowledge

## 最佳实践

- 面向非工程用户时，Project Copilot 应优先使用“项目秘书”语言，而不是工程工作流语言。
- 首次体验应先接收完整项目方案，再只追问缺失项，避免默认发问卷或让用户手工填写空模板。
- 复盘输出应列表化、短句化，避免大段文字。
- 事实、假设、计划和决策必须分层存放，否则 `continue` 与复盘类流程容易污染长期记忆。
- `continue` 的不确定性判断应只读取假设层，不要把运行态的占位文本误判成待确认事项。
- 维护型动作如同步和复盘的归档，应优先写入状态文件或事件归档，不要自动进入 `MEMORY.md`。
- 初始化和接管入口只应向 `MEMORY.md` 写短事件摘要，不要写入用户原始输入全文。
- 已确认决策应只由 `DECISIONS.md` 承载，不要再复制到 `MEMORY.md`。

## 参考项目

- 当前阶段继续使用 `ai-recruitment` 作为真实项目验证案例。

## 产品认知

- Codex 负责开发实现，Project Copilot 负责记录为什么这样做。
- 成功标准不是功能更多，而是用户 30 天后仍能回答项目为什么做、如何演化、哪些需求被放弃。

## 重要经验

- `.ai/MEMORY.md` 不应堆重复同步流水，应沉淀成可读的项目历史。
- `HYPOTHESES.md` 是未确认判断的缓冲层，不应直接升级为事实或决策。
- 用户文档和生成模板要统一使用“分层记忆”术语，避免把 `MEMORY.md` 叙述成单一长期仓库。
- 历史文本清扫时应优先保留里程碑和结构变化，删掉批量同步痕迹这类低信息密度记录。
- 路线图和状态页也要同步记忆边界收口，否则新的边界只会停留在聊天和代码里。
- `.ai/history/` 适合按月归档，单次时间戳文件会降低可回看性。
- 复盘归档最好固定成少量字段，便于月度对比和检索。
- 复盘里的决策窗口不宜过长，最近 3 条通常足够支撑判断。
- 提取复盘决策时应按决策块解析，不要按单行截断。
- “最近 3 条”应理解为最近 3 条完整决策，不是排除倒数第二条。
- 时间轴更适合固定栏目输出，而不是把所有事件平铺成列表。
- 辅助指标文件如果不能可靠派生，就不要把它放进核心事实层；宁可降级成快照，也不要让它和事实源竞争。
- Project Copilot 的长期价值在于帮助 Codex 记住“为什么”，普通代码修改和测试记录应交给 Git。
- Session Memory 可以降低长期记忆污染：开始时读上下文，过程中只保留候选事件，结束时确认后写入。
- ADR 比单一决策流水更适合 AI Coding，因为它能把背景、原因、取舍和影响拆成可引用的小记录。
- Validation 应跟随长期记忆写入自动刷新；Session 候选和未确认假设不应触发派生报告更新。
- P0 级边界收敛应优先移除与 Codex/Git 重合的自动化动作，再优化用户可见体验。
- Memory View 的价值不在于包装读取单个 Markdown 文件，而在于把 Charter、ADR、Status、Roadmap、Session 和 drift signals 组合成可判断的记忆健康视图。
